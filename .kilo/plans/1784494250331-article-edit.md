# План: Редактирование статьи в любом статусе

## Текущее состояние

- Кнопка «Редактировать» в `author-articles.tsx:578` ведёт на `/articles/new?id=${id}`
- Визард уже загружает статью по `?id=` и заполняет все поля (`article-wizard-client.tsx:349-461`)
- **Нет PATCH-роута** — `app/api/articles/[id]/route.ts` содержит только GET
- **Нет сохранения** — `handlePublish` всегда делает POST (создаёт новую), даже когда `createdArticleId` уже задан
- Модель `updateArticle` в `lib/models.ts:366` готова, но ни один API-роут её не вызывает
- Прогресс-бар визуальный, не кликабельный
- Валидация шагов (`validateStep`) не различает create/edit режим

## Целевое поведение

### Для всех статусов

| Текущий статус   | «Сохранить»                               | «Опубликовать»                                             |
| ---------------- | ----------------------------------------- | ---------------------------------------------------------- |
| `draft`          | Остаётся `draft`                          | `pending_review` (оплачено) или `pending_payment`          |
| `pending_review` | Переходит в `draft` (отзыв с модерации)   | `pending_review` или `pending_payment`                     |
| `published`      | Переходит в `draft` (снятие с публикации) | Сохраняется как `draft`, затем стандартный поток модерации |
| `archived`       | Переходит в `draft` (разархивация)        | Сохраняется как `draft`, затем стандартный поток           |

### Шаги визарда в режиме редактирования

| Шаг                    | Поведение                                                                                                                  |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 1-5 (Отрасль → Ссылки) | Поля **заполнены** из БД. Можно изменить отрасль/подсектор/категорию/экспертизу/кросс-ссылки. Валидация — как при создании |
| 6 (Заголовок)          | Заголовок, введение, slug — из БД. Редактируемы                                                                            |
| 7 (Текст)              | Секции загружены из `sectionsText`. Можно добавлять/удалять/редактировать секции                                           |
| 8 (FAQ)                | FAQ загружен. Редактируем                                                                                                  |
| 9 (Чеклист)            | Чеклист загружен. Редактируем                                                                                              |
| 10 (Данные)            | TL;DR, Key Facts, Definition, Featured Snippet, Problem→Solution→Result, HowTo — из БД                                     |
| 11 (Предпросмотр)      | Полный предпросмотр с текущими данными                                                                                     |
| 12 (Публикация)        | Кнопки: «Сохранить черновик» + «Опубликовать»                                                                              |

## Этап 1: PATCH /api/articles/[id]

### Файл: `app/api/articles/[id]/route.ts`

Добавить экспорт `PATCH`:

```ts
export async function PATCH(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  // 1. Авторизация (verifyToken)
  // 2. Загрузка статьи (getArticleById)
  // 3. Проверка: article.expertId === payload.id (только автор)
  // 4. Валидация тела: updateArticleFullSchema (все поля optional)
  // 5. Определение целевого статуса:
  //    - Если body.status передан явно — использовать его
  //    - Если нет — оставить текущий статус
  // 6. Статус-машина:
  //    published → publish()? → сохраняем как draft (снимаем с публикации)
  //    pending_review → сохраняем как draft (отзываем)
  //    archived → сохраняем как draft (разархивируем)
  // 7. updateArticle(id, { ...body, status: targetStatus })
  // 8. 200 + { article }
}
```

### Статус-машина (хелпер)

```ts
function transitionStatus(current: string, requested?: string): string {
  // Явный запрос статуса имеет приоритет
  if (
    requested &&
    ["draft", "pending_review", "pending_payment"].includes(requested)
  ) {
    return requested;
  }
  // published/archived/pending_review при сохранении → draft
  if (["published", "archived", "pending_review"].includes(current)) {
    return "draft";
  }
  // draft остаётся draft
  return current;
}
```

### Схема валидации

Создать `updateArticleFullSchema` — все поля из `articleSchema` (POST), но optional:

```ts
const updateArticleFullSchema = z
  .object({
    title: z.string().min(10).max(200).optional(),
    description: z.string().min(40).max(600).optional(),
    content: z.string().min(100).max(150000).optional(),
    slug: z
      .string()
      .min(2)
      .max(200)
      .regex(/^[a-z0-9-]+$/)
      .optional(),
    status: z.enum(["draft", "pending_review", "pending_payment"]).optional(),
    // ... все остальные поля — optional
  })
  .partial();
```

Поместить схему в `lib/validation/articles.ts` (рядом с `updateArticleSchema`).

```ts
const [saving, setSaving] = useState(false);

async function handleSaveDraft() {
  setSaving(true);
  const token = localStorage.getItem("token");
  const body = buildRequestBody(data); // та же сборка, что в handlePublish
  body.status = "draft"; // явно сохраняем как черновик

  const res = await fetch(`/api/articles/${createdArticleId}`, {
    method: "PATCH",
    headers: {
      "Content-Type": "application/json",
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
    body: JSON.stringify(body),
  });

  if (res.ok) {
    toast.success("Черновик сохранён");
    localStorage.removeItem(STORAGE_KEY);
  } else {
    toast.error("Ошибка сохранения");
  }
  setSaving(false);
}
```

#### 2.3. Исправление handlePublish

В `handlePublish` (строка 606), когда `createdArticleId` существует:

**Важно**: для `published` статей, при нажатии «Опубликовать», PATCH-роут должен перевести в `pending_review`/`pending_payment`. Добавить в body флаг:

```ts
body._publishAction = true;
```

И в PATCH-роуте: если `_publishAction === true`, то целевой статус = `pending_review` (если оплачено) или `pending_payment`.

## Этап 0: Исправление hydration-ошибки

**Проблема**: внешний HTTPS-прокси добавляет `data-heading-tag="H1"` / `data-heading-tag="H2"` на серверный рендер, но клиентский рендер этих атрибутов не содержит → hydration mismatch.

### Файл: `components/catalog-client.tsx`

Добавить `suppressHydrationWarning` на `<h1>` (строка 199) и `<h2>` (строка 247):

```tsx
// Строка 199:
<h1 className="..." suppressHydrationWarning>

// Строка 247:
<h2 className="..." suppressHydrationWarning>
```

`suppressHydrationWarning` в React 18+ подавляет предупреждения о несовпадении как текста, так и атрибутов на элементе. Атрибут `data-heading-tag` добавляется прокси-сервером, мы не можем его контролировать.

## Риски

| Риск                                                                 | Решение                                                                                                               |
| -------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Снятие published статьи с публикации теряет SEO-позиции              | Показать явный alert перед сохранением. Пользователь осознанно соглашается                                            |
| Одновременное редактирование с двух вкладок                          | `updatedAt` в БД. При конфликте — последний сохранивший перезаписывает. Допустимо для v1                              |
| Slug меняется при редактировании заголовка                           | Slug редактируется вручную на шаге 6. Не авто-генерируется                                                            |
| Импортированная статья (industryId="none") не валидируется на шаге 1 | Уже решено: при загрузке через `?importId=` empty values конвертируются. Для `?id=` (edit) — значения из БД настоящие |
| Поломка localStorage при переключении create/edit                    | При загрузке статьи через `?id=` очищать `STORAGE_KEY` или не перезаписывать существующий                             |
